7x Resource Duplication

Each ChannelHandler gets its own TelegrafHTTPEventProcessor with independent HTTP connection pool, write buffer, and retry logic.

Retry Storms

When InfluxDB is unreachable, all 7 processors retry independently — 7x the error logs, 7x the backoff timers, 7x the reconnection attempts.

Wasteful Control Channel

Control (ch 0) handles auth/keepalive messages with no market data, yet still gets an InfluxDB processor it will never use.

7x Shutdown Flush

close_processors() called per handler in the finally block. Each flush independently drains its write buffer and closes its HTTP session.

Single Shared Instance

One TelegrafHTTPEventProcessor created in run() and injected into 6 handlers (all except Control). Single connection pool, single write buffer.

Unified Retry

During InfluxDB outages, only one processor retries. No parallel retry storms — clean, predictable backoff behavior.

Control Channel Skipped

Control handler (ch 0) handles auth/keepalive only — no market data, no InfluxDB processor needed. Explicit skip with if ch == Channels.Control: continue.

Clean Shutdown

Handlers close their own processors (Redis only). The shared InfluxDB processor is closed once in the finally block — single flush, single HTTP session teardown.
